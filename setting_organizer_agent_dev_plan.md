# SillyTavern 设定整理器 Agent 开发计划

## 1. 开发总目标

开发一个 SillyTavern 第三方前端扩展，用于把用户粘贴的设定文本整理为：

- 角色卡草稿
- 世界书条目草稿
- 世界书关键词
- 警告信息
- 可导出的 JSON

第一版只追求安全可用，不追求全功能。

核心流程：

```text
粘贴设定文本
-> 调用当前 SillyTavern 模型
-> 获取严格 JSON
-> 校验和修复结果
-> 展示角色卡与世界书草稿
-> 用户手动编辑
-> 导出 JSON
-> 可选创建新角色 / 新世界书
```

## 2. 第一版关键原则

- 默认不覆盖任何已有数据。
- 默认不删除任何已有数据。
- 默认不修改全局世界书。
- AI 不能直接写入 SillyTavern。
- 写入前必须经过预览和用户确认。
- 模型输出异常时不能导致扩展崩溃。
- 先实现只整理与导出，再实现导入。
- 内部草稿格式必须和 SillyTavern 写入格式分离。
- 第一版只承诺“备份 + 失败状态报告 + 可手动恢复依据”，不承诺完整自动回滚，除非后续实际实现并通过验证。

## 2.1 MVP 分层

为降低 Agent 一次性交付风险，第一版拆成三个验收层级。

### MVP-A：整理与导出闭环

必须完成：

- 扩展骨架
- 粘贴文本
- AI 调用
- JSON 解析
- Schema 校验
- 角色草稿
- 世界书草稿
- 人工编辑
- JSON 导出

验收目标：

```text
粘贴文本
-> AI 返回 JSON
-> 校验
-> 人工编辑
-> 导出
```

### MVP-B：安全新建闭环

在 MVP-A 稳定后实施。

必须完成：

- 备份
- 导入确认
- 创建新世界书
- 创建新角色
- 导入失败状态报告
- 导入前后旧数据不变验证

验收目标：

```text
已编辑草稿
-> 创建备份
-> 用户确认
-> 新建角色 / 世界书
-> 证明旧数据未变化
```

### MVP-C：输入来源与兼容增强

在 MVP-B 稳定后实施。

包含：

- 当前聊天读取
- Android / Termux 兼容性强化
- 更完整的移动端 UI 验证

## 3. Agent 执行规则

Agent 必须按阶段推进，不允许跳阶段开发高级功能。

每个阶段必须完成：

```text
1. 阅读现有项目结构
2. 确认当前阶段依赖
3. 实现最小功能
4. 做手动或自动验证
5. 记录已完成内容和遗留问题
```

写入文件前，Agent 必须先说明：

- 计划新增哪些文件
- 计划修改哪些文件
- 每个文件的作用
- 是否影响已有功能

获得用户明确许可后，才能执行写入。

## 4. 第一版禁止范围

第一版禁止实现：

- 自动持续更新
- 多项目同步
- RAG / Data Bank 管理
- 关系图
- 完整版本差异
- 高级语义冲突推理
- PNG 角色卡封装
- 自动覆盖已有角色或世界书

## 7. 阶段计划

### 阶段 0：项目侦察

目标：确认当前项目环境，而不是直接写代码。

任务：

- 检查当前目录是否已有 SillyTavern 扩展代码。
- 查找 `manifest.json`、`index.js`、`README.md`、`package.json`。
- 确认是空目录、新项目，还是已有项目。
- 只读检查 SillyTavern 第三方扩展结构。
- 输出实施建议。

验收标准：

- Agent 能说明当前目录现状。
- Agent 能确认下一步是新建扩展还是改造已有扩展。
- 没有进行任何写入操作，除非用户明确授权。

### 阶段 1：扩展骨架

目标：创建能被 SillyTavern 加载的最小扩展。

功能：

- `manifest.json`
- `index.js`
- `style.css`
- 基础扩展入口
- 扩展面板
- 文本输入框
- “开始分析”按钮
- 结果区域
- 基础调试日志

验收标准：

- 扩展可以在 SillyTavern 扩展列表中加载。
- 打开面板不会报错。
- 输入文本后点击按钮，能在结果区显示占位结果。
- 刷新页面后扩展仍能正常加载。

### 阶段 1A：SillyTavern API 探针

目标：在调用模型、创建角色或写入世界书前，先确认当前 SillyTavern 版本可用的扩展接口。

任务：

- 确认扩展入口和加载生命周期。
- 确认可用的设置保存方式。
- 确认当前模型调用方式。
- 确认当前聊天读取方式。
- 确认角色创建能力。
- 确认世界书创建能力。
- 确认是否存在官方接口、稳定辅助函数或只能访问内部对象。

输出兼容性矩阵：

```text
能力 | 可用性 | 接口位置 | 是否稳定 | 备注
扩展加载 | 是/否 | ... | ... | ...
设置保存 | 是/否 | ... | ... | ...
模型调用 | 是/否 | ... | ... | ...
聊天读取 | 是/否 | ... | ... | ...
角色创建 | 是/否 | ... | ... | ...
世界书创建 | 是/否 | ... | ... | ...
```

验收标准：

- 不进行业务写入。
- 所有 SillyTavern 内部对象访问方案都有记录。
- 如果某项能力不可用，后续任务必须降级或标记阻塞。

### 阶段 2：本地 UI 与状态管理

目标：先把前端流程跑通，不依赖模型。

功能：

- 粘贴文本输入。
- 整理目标选择：角色卡、世界书。
- Token 预算模式：轻量、标准、长篇、自定义。
- 本地设置保存。
- 分析状态：空闲、分析中、成功、失败。
- 错误提示区域。

验收标准：

- 用户输入不会因切换标签页或刷新立即丢失。
- 设置可以保存到浏览器本地存储或 SillyTavern 扩展设置。
- UI 在桌面和手机窄屏下可用。
- 按钮状态不会重复触发并发分析。

### 阶段 3：模拟分析结果

目标：在接入 AI 前，先完成结果展示和编辑。

功能：

- 使用固定 mock JSON 生成角色草稿。
- 使用固定 mock JSON 生成世界书草稿。
- 展示总览。
- 展示角色卡字段。
- 展示世界书条目。
- 支持编辑字段。
- 支持删除草稿条目。
- 支持启用 / 禁用世界书条目。
- 支持修改关键词。

验收标准：

- mock 数据能完整展示。
- 用户编辑后，内部状态同步更新。
- 删除条目不会影响其他条目。
- 世界书关键词可以新增、删除、修改。
- 页面切换不会丢失编辑结果。

### 阶段 4：JSON Schema 与校验

目标：保证模型或 mock 输出不会直接污染内部状态。

功能：

- 定义分析结果 Schema。
- 定义角色草稿 Schema。
- 定义世界书草稿 Schema。
- 实现 JSON 解析。
- 实现字段补全。
- 实现类型修正。
- 实现非法结果拦截。
- 实现错误码。

错误码第一版：

```text
E001 模型调用失败
E002 模型输出不是合法 JSON
E003 JSON Schema 校验失败
E004 结果为空
E005 Token 超过预算
E006 导出失败
E007 创建备份失败
E008 创建角色失败
E009 创建世界书失败
E010 当前 SillyTavern 接口不兼容
E011 旧数据完整性校验失败
E012 当前聊天读取失败
```

验收标准：

- 非 JSON 文本不会导致页面崩溃。
- 缺少可补全字段时能补默认值。
- 关键字段类型错误时能报错。
- 错误信息能展示给用户。
- 调试日志不包含 API Key 或敏感认证信息。

Schema 异常处理规则：

| 场景 | 处理方式 |
| --- | --- |
| 顶层不是对象 | 拒绝，显示 `E003` |
| `characters` 缺失 | 补空数组 |
| `lorebookEntries` 缺失 | 补空数组 |
| `characters` 和 `lorebookEntries` 均为空 | 拒绝，显示 `E004` |
| `warnings` 缺失 | 补空数组 |
| `name` 为空 | 允许进入编辑，增加警告，禁止直接导入 |
| `title` 为空 | 允许进入编辑，增加警告，禁止直接导入 |
| `keys` 为字符串 | 转为单元素数组，并增加警告 |
| `keys` 为空数组 | 允许进入编辑，增加警告，禁止直接导入世界书 |
| `confidence` 缺失 | 补 `0.8` |
| `confidence` 超出 0 到 1 | 截断到范围内，并增加警告 |
| 未识别字段 | 内部草稿保留，导出到 SillyTavern 兼容格式时剔除 |

### 阶段 5：Prompt 与 AI 调用

目标：调用 SillyTavern 当前连接的大模型，生成严格 JSON。

Prompt 要求：

- 明确要求只输出 JSON。
- 明确 `schemaVersion`。
- 明确不要输出 Markdown 代码块。
- 明确无法判断时留空或写入 warnings。
- 明确区分永久设定和动态剧情状态。
- 明确不要编造未出现的信息。

Prompt 版本：

```text
extract-setting-v0.1.0
```

验收标准：

- 输入简单角色设定，能生成至少 1 个角色草稿。
- 输入世界观设定，能生成至少 1 个世界书条目。
- 模型返回非 JSON 时显示 `E002`。
- 模型调用失败时显示 `E001`。
- 用户可以重新分析。

### 阶段 6：Token 粗估与基础警告

目标：给用户基础风险提示，不做复杂语义推理。

Token 粗估规则：

```text
中文：1 个汉字约 1 token
英文：4 个字符约 1 token
混合文本：总字符数 / 2 作为保守估算
```

第一版警告：

- 输入过长
- 角色名称为空
- 世界书标题为空
- 世界书内容为空
- 世界书关键词为空
- 世界书关键词少于 2 个字符
- 世界书关键词重复
- 常驻条目过多
- 单条世界书内容过长

验收标准：

- 警告能显示在总览和对应条目上。
- 警告不会阻止用户编辑。
- 严重错误才阻止导出或导入。
- Token 估算结果能随编辑变化更新。

### 阶段 7：JSON 导出

目标：先实现安全导出，不急于写入 SillyTavern。

导出类型：

- 内部完整草稿 JSON
- 角色草稿 JSON
- 世界书草稿 JSON
- SillyTavern 兼容角色 JSON
- SillyTavern 兼容世界书 JSON

需要实现 adapter：

```text
InternalCharacterDraft -> SillyTavernCharacterExport
InternalLorebookDraft -> SillyTavernWorldInfoEntry
```

验收标准：

- 导出的 JSON 可被 `JSON.parse` 解析。
- 内部元数据不会误写入 SillyTavern 兼容格式。
- 用户编辑后的内容能正确导出。
- 导出失败显示 `E006`。

### 阶段 8：备份设计

目标：在任何写入 SillyTavern 前，先建立备份能力。

备份内容：

```json
{
  "backupVersion": "0.1.0",
  "createdAt": "",
  "operation": "",
  "sillyTavernVersion": "",
  "sourceDraft": {},
  "targetInfo": {},
  "beforeState": {},
  "afterState": null
}
```

验收标准：

- 备份成功后才允许导入。
- 备份失败显示 `E007`。
- 用户能看到备份时间和备份标识。
- 第一版不承诺完整自动回滚。
- 失败时必须显示已完成步骤、未完成步骤、可能影响范围和备份标识。

### 阶段 9：创建新世界书

目标：只创建新世界书，不修改已有世界书。

功能：

- 导入确认窗口。
- 显示即将创建的世界书名称。
- 显示条目数量。
- 显示关键词风险。
- 创建新世界书。
- 写入启用条目。
- 保存导入日志。

验收标准：

- 默认创建新世界书。
- 不修改全局世界书。
- 不修改已有世界书。
- 导入前后必须对比已有世界书摘要，证明旧世界书未变化。
- 失败时显示 `E009`。
- 导入结果显示成功创建的对象名称或 ID。

### 阶段 10：创建新角色

目标：只创建新角色，不覆盖当前角色。

功能：

- 将角色草稿转换为 SillyTavern 角色格式。
- 创建新角色。
- 可选绑定新建世界书。
- 导入确认窗口展示所有变化。
- 保存导入日志。

验收标准：

- 默认创建新角色。
- 不覆盖当前角色。
- 导入前后必须对比已有角色摘要，证明旧角色未变化。
- 失败时显示 `E008`。
- 创建成功后用户可以在 SillyTavern 中看到新角色。
- 如果绑定世界书失败，角色本身不应被静默视为完整成功。

### 阶段 11：当前聊天读取

目标：在 MVP 稳定后，再支持从当前聊天提取设定。

功能：

- 读取当前聊天消息。
- 用户选择读取范围：最近 20 条、最近 50 条、全部、手动选择。
- 显示预计输入长度。
- 发送给分析流程。

验收标准：

- 聊天读取失败不会影响粘贴文本模式。
- 超长聊天会提示 Token 风险。
- 不自动读取隐私内容，必须用户主动选择。

### 阶段 12：基础重复检测

目标：只做规则级检测，不做复杂语义推理。

检测内容：

- 同名角色
- 同标题世界书
- 完全相同关键词
- 完全相同世界书正文
- 过短关键词
- 泛化关键词，例如“他”“她”“这里”“世界”“人类”

验收标准：

- 重复项能在警告页展示。
- 用户可以选择保留或删除。
- 检测不自动删除内容。

## 8. 每阶段输出格式

Agent 每完成一个阶段，应输出：

```text
完成内容：
- ...

修改文件：
- ...

验证结果：
- ...

- ...

- ...
```

## 9. 测试样例

### 9.1 简单角色设定

```text
林月是一名银月教会的见习祭司，性格温和但有些固执。她擅长治疗术，害怕黑暗。故事开始时，她正在寻找失踪的导师。
```

期望：

- 生成 1 个角色。
- 角色名为林月。
- `personality` 不为空。
- `scenario` 不为空。
- 至少生成 1 个世界书条目：银月教会。

### 9.2 世界观设定

```text
艾泽兰大陆分为北境、王都和灰雾边境。银月教会掌管治疗与净化仪式，黑曜商会控制跨境贸易。灰雾边境常年出现魔物。
```

期望：

- 生成多个世界书条目。
- 每个条目有 `title`、`keys`、`content`。
- 关键词不应只有“大陆”“教会”“商会”这类过泛词。

### 9.3 错误 JSON

```text
这是模型解释，不是 JSON。
```

期望：

- 显示 `E002`。
- 页面不崩溃。
- 可以重新分析。

### 9.4 重复设定

```text
银月教会负责治疗。
银月教会负责治疗与净化。
银月教会是治疗组织。
```

期望：

- 生成一个合并后的世界书条目，或提示重复风险。
- 不应生成三条几乎相同的条目。

## 10. 第一版最终验收标准

### MVP-A 完成条件

1. 扩展能在 SillyTavern 中加载。
2. 用户能粘贴设定文本。
3. 用户能调用当前模型分析。
4. 模型输出能被解析和校验。
5. 非 JSON 输出不会导致崩溃。
6. 能生成角色草稿。
7. 能生成世界书草稿。
8. 用户能编辑生成结果。
9. 用户能导出 JSON。

### MVP-B 完成条件
