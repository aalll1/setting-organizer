# SillyTavern 设定整理器 QA 评审意见

## 1. 评审范围

当前目录没有实际扩展代码，只有规划类 Markdown 文档。因此本次 QA 评审对象是：

- `sillytavern_setting_organizer_dev_plan.md`
- `setting_organizer_agent_dev_plan.md`
- `setting_organizer_task_cards.md`
- `setting_organizer_doc_changelog.md`

本次评审属于规格评审和任务卡评审，不是代码缺陷审查。

## 2. 总体结论

文档已经具备启动开发的基础，但在交给 Agent 执行前，应先修正以下高风险点：

- 错误码定义不一致。
- 备份和回滚承诺口径不一致。
- MVP 边界仍偏大。
- 缺少 SillyTavern API 探针任务。
- Schema 校验规则不够可测。
- 导入任务缺少“已有数据不变”的验证方法。
- 测试样例覆盖不足。

建议先修订文档，再进入扩展骨架开发。

## 3. 主要问题

### 3.1 错误码不一致

问题：

- 原始计划中 `E005` 是世界书写入失败，`E006` 是角色创建失败。
- Agent 版计划中 `E006` 是导出失败，`E008` 是创建角色失败，`E009` 是创建世界书失败。

风险：

- 后续实现可能出现错误码映射混乱。
- UI、日志、测试用例无法稳定断言。

建议：

- 单独建立统一错误码规范。
- 所有文档引用同一套错误码。

### 3.2 备份和回滚承诺不一致

问题：

- 原始计划要求“错误回滚”“导入失败时原数据不受影响”。
- Agent 版计划写明“不承诺完整自动回滚，除非实际实现并验证”。

风险：

- 开发 Agent 可能按过强承诺实现，导致复杂度失控。
- 用户可能误以为第一版支持事务式回滚。

建议：

- 第一版统一表述为“创建备份 + 失败状态报告 + 可手动恢复依据”。
- 不承诺完整自动回滚，除非后续明确实现并通过测试。

### 3.3 MVP 边界仍偏大

问题：

- 当前最终验收把 AI 调用、编辑、导出、备份、新建世界书、新建角色、Android / Termux 可用性放在同一层。
- 当前聊天读取虽然放在后续阶段，但仍出现在第一版增强项中。

风险：

- Agent 一次性交付范围过大。
- 失败时难以判断是核心链路问题，还是导入 / 环境兼容问题。

建议：

- 拆分为 MVP-A、MVP-B、MVP-C。
- MVP-A：粘贴文本、AI、校验、编辑、导出。
- MVP-B：备份、新建世界书、新建角色。
- MVP-C：当前聊天读取、Android / Termux 兼容性强化。

### 3.4 缺少 SillyTavern API 探针任务

问题：

- 任务卡中 TC-06 直接要求调用当前模型。
- TC-11、TC-12 直接要求创建世界书和角色。
- 但没有前置任务确认当前 SillyTavern 版本暴露哪些扩展 API。

风险：

- 后续实现建立在未验证接口上。
- Agent 可能直接访问内部对象，造成版本兼容风险。

建议：

- 在 TC-01 后新增 `TC-01A API 探针`。
- 该任务只做只读确认和最小探测，不做业务写入。
- 输出兼容性矩阵。

### 3.5 Schema 校验策略不够可测

问题：

- 文档写了“字段补全、类型修正、非法结果拦截”，但没有定义哪些情况可补、哪些必须拒绝。

风险：

- 不同 Agent 对同一异常输出处理不一致。
- 测试无法稳定断言。

建议：

- 增加“可修复 / 不可修复”规则表。
- 明确 `characters`、`lorebookEntries`、`name`、`keys`、`confidence` 等字段的处理规则。

### 3.6 导入验收缺少旧数据不变验证

问题：

- 文档要求不覆盖、不修改已有世界书。
- 任务卡没有要求导入前后对比已有对象。

风险：

- 导入功能可能误改旧数据但未被测试发现。

建议：

- TC-11、TC-12 增加导入前后快照对比。
- 验收必须证明旧角色、旧世界书摘要未变化。

### 3.7 测试样例覆盖不足

问题：

- 当前样例覆盖简单角色、世界观、错误 JSON、重复设定。
- 缺少空输入、半截 JSON、Markdown 包裹 JSON、字段类型错误、超长输入、备份失败、导入失败。

风险：

- 主要失败路径没有测试基线。

建议：

- 扩充测试样例清单。
- 每个阶段绑定最小验证样例。

## 4. 建议优先级

### P0 必须先修

- 统一错误码。
- 统一备份 / 回滚口径。
- 新增 API 探针任务。
- 拆分 MVP 验收。

### P1 应在开发前补齐

- Schema 可修复 / 不可修复规则。
- 导入前后旧数据不变验证。
- 测试样例扩充。

### P2 后续持续完善

- 自动化测试矩阵。
- Android / Termux 专项兼容清单。
- SillyTavern 版本兼容记录。

## 5. 建议后的开发准入条件

进入实际开发前，文档应至少满足：

1. 所有错误码在文档中一致。
2. 第一版不承诺完整自动回滚。
3. MVP-A / MVP-B / MVP-C 边界明确。
4. 任务卡包含 API 探针。
5. Schema 异常处理规则可测试。
6. 导入任务包含旧数据不变验证。
7. 测试样例覆盖主要失败路径。
